## 「気づいたら 10 分止まってた」問題

Claude Code を使い込むほど、セッションは並列になっていきます。リポジトリA で機能実装、リポジトリB でバグ調査、worktree でリファクタ——ここまでは快適なのですが、並列数が増えると必ずこうなります。

1. セッションを走らせて、別の作業に移る
2. その裏でセッションが権限確認(`Do you want to allow...?`)で止まる
3. **気づかない**
4. 10分後に見に行って「ずっと待ってたのか…」となる

エージェントの実行時間より、**人間が「待たれていることに気づくまでの時間」がボトルネック**になるわけです。ターミナルのタブを目視で巡回するのは並列数3を超えたあたりで破綻します。

この問題を解決するために、Claude Code の全セッションを監視して「入力待ちになった瞬間だけ」フローティングウィンドウで知らせる macOS アプリ **AgentManager** を作りました。この記事では、その裏側の実装——特に「どうやってセッション状態を検知するか」「どうやって正しいターミナルにジャンプするか」を書きます。

## 全体アーキテクチャ: デーモンなし、ソケットなし、全部ファイル

構成は徹底的に単純にしました。

```
Claude Code
  └─ hook 発火(SessionStart, PreToolUse, Notification, Stop, ...)
      └─ コンパイル済み hook バイナリ(Swift)
          └─ ~/.claude/agent-manager/sessions/<session_id>.json を upsert
              └─ GUI アプリが FSEvents でディレクトリを監視
```

Claude Code には [hooks](https://docs.anthropic.com/en/docs/claude-code/hooks) という仕組みがあり、セッションのライフサイクルイベント(セッション開始、ツール実行前後、通知、停止など)で任意のコマンドを実行できます。AgentManager はこれを9イベントすべて購読し、受け取ったイベントをセッションIDごとの JSON ファイルに畳み込みます。GUI 側はそのディレクトリを FSEvents(`DispatchSource`)で監視するだけ。

デーモンも Unix ソケットも IPC も使っていません。ファイルベースにした理由:

- **hook プロセスは毎回起動して即死ぬ**。常駐相手との接続管理が要らないファイル追記(正確には置換)が一番壊れにくい
- GUI が落ちていてもイベントは失われない(ファイルに残る)
- デバッグが楽。`cat ~/.claude/agent-manager/sessions/*.json` で全状態が見える

ただしファイルベースには並行性の罠があります。サブエージェントが並列で走ると同一セッションの JSON に複数の hook プロセスが同時に書きに来るため、**セッション単位の `flock` でロック**し、tmp ファイルに書いてから `rename(2)` でアトミックに置換しています。これを入れる前は lost update でサブエージェントの終了イベントが消えることがありました。

## 一番むずかしいのは「waiting」の判定

「実行中」「完了」は簡単です。難しいのは「**人間の入力を待っている**」の判定で、これがこのアプリの存在意義そのものです。

hook イベントとセッション状態のマッピングは、純粋関数の状態機械(reducer)として実装しています。テストが書きやすいからです。基本はこうです:

| hook イベント | 状態 |
|---|---|
| `SessionStart` | idle |
| `UserPromptSubmit` / `PreToolUse` / `PostToolUse` | processing |
| `Stop` | done |
| `Notification` | **中身による(ここが本題)** |

### Notification は種類で分岐する

「入力待ち」は `Notification` イベントで届きますが、Notification には種類があり、全部を waiting にすると誤爆します。

- `permission_prompt`(権限確認)、`elicitation_dialog`(質問ダイアログ)→ **waiting**
- `idle_prompt`(しばらく操作がないときの通知)→ waiting にしない。これを waiting にすると「完了して放置していたセッション」が確認待ちに化けて、狼少年アプリになる
- 認証成功などその他の通知 → 状態維持

### Notification が来ないのに待っているケース

さらに厄介なのが、**ユーザーに選択を求めているのに `Notification` を発火しないケース**です。具体的には `AskUserQuestion`(選択肢の質問)や `ExitPlanMode`(プラン承認)。これらは画面上では明確に「人間の番」なのに、通知イベントは飛んで来ません。

そこで、これらのツールは `PreToolUse`(ツール実行直前のイベント)の時点で waiting に倒すという特別扱いをしています。「このツールの実行 = 人間への質問」なので、実行前イベントがそのまま入力待ち通知として使えるわけです。

### サブエージェントで「早すぎる done」を防ぐ

Claude Code はサブエージェントを並列に走らせることがあります。このときメインの `Stop` が先に来ても、サブエージェントがまだ動いていれば仕事は終わっていません。素朴に `Stop → done` とすると「完了と表示されたのにまだガリガリ動いている」が起きます。

そのため状態には稼働中サブエージェントの ID 集合を持たせて、

- `SubagentStart` で集合に追加
- メインの `Stop` が来ても集合が空でなければ processing を維持
- 最後の `SubagentStop` で done

としています。

## 「どのターミナルで動いているか」はプロセスツリーで判定する

ジャンプ機能(後述)のためには、各セッションが**どのアプリのターミナルで動いているか**を記録する必要があります。

最初は環境変数(`__CFBundleIdentifier` や `ITERM_SESSION_ID`)を見ていたのですが、これは罠でした。**環境変数は子プロセスに継承される**ので、たとえば iTerm2 から Android Studio を起動すると、Android Studio の統合ターミナルで走る Claude Code にも `ITERM_SESSION_ID` が付いてきます。「iTerm2 のセッション」と誤判定するわけです。

現在は hook バイナリが `ps -axww -o pid=,ppid=,comm=` のスナップショットを1回だけ取り、プロセスツリーを遡って**最も近い祖先の `.app` バンドルの Info.plist から `CFBundleIdentifier` を読む**方式にしています。プロセスの親子関係は継承で汚染されないので、これで統合ターミナル問題が解決しました。

細かい罠も2つ紹介します。

- **PID 再利用**: セッションの生死判定に PID を使うと、OS が PID を再利用したときに「死んだセッションが生きている」ことになる。`kill(pid, 0)` の生存確認に加えて、プロセスの起動時刻(`ps -o lstart=`)を記録時と照合して同一プロセスであることを確認しています
- **ロケール**: その `lstart` の文字列表現はロケールで変わります。記録側と照合側でロケールが違うと、生きているセッションを孤児と誤判定して消してしまう。両側で `LC_ALL=C` を固定して解決

## ターミナルへのジャンプは「ホストごとに全部違う」

一覧のセッションをクリックすると、そのセッションが動いているターミナルへ移動します。この機能、ホストアプリごとに実現手段がバラバラで、実装は完全に地道な各個撃破でした。

| ホスト | 手段 | 精度 |
|---|---|---|
| iTerm2 | AppleScript。セッション固有の GUID(`ITERM_SESSION_ID`)で全 window/tab/session を走査して `select` | **ペイン単位**(最強) |
| Terminal.app | AppleScript。GUID が無いので、セッションの PID から `ps -o tty=` で **TTY を導出し、各タブの tty プロパティと照合** | タブ単位 |
| Ghostty 1.3+ | AppleScript。公開されているのは id / name / working directory のみ。**cwd がちょうど1件一致したときだけ**フォーカス(0件/複数件ならアプリ前面化に留めて誤爆回避) | タブ/ペイン単位(条件付き) |
| VS Code / Cursor / Zed | 各 CLI(`code` 等)に `--reuse-window <cwd>` | プロジェクトウィンドウ |
| Warp / kitty / WezTerm / Alacritty | `open -b` で前面化のみ | アプリ単位 |
| tmux | **意図的に非対応**(検知はする) | — |

tmux を非対応にしているのは技術的にできないからではなく、**誤爆するくらいなら飛ばない方がマシ**だからです。tmux はサーバがデーモン化するので、プロセスツリーを遡っても「いまユーザーが見ているクライアントの端末」に辿り着けません。間違ったウィンドウに飛ぶ機能は、無い機能より体験が悪い。

あと、AppleScript の実行(`osascript`)は専用のシリアルキューで直列化しています。ジャンプを連打されたときに並行実行されると挙動が壊れるのと、メインスレッドで実行するとアプリ全体がビーチボール化するためです。

## 他人の settings.json を書き換えるという緊張感

hook の登録先は `~/.claude/settings.json`、つまり**ユーザーの設定ファイル**です。ここを機械的に上書きするのは、開発者のドットファイル環境を壊す最短ルートでもあります。実際に配慮したこと:

- **`hooks` キーの値だけを置換する**。JSON 全体をパースして書き戻すと、キー順・インデント・末尾改行が全部変わってしまう。そこで balanced-brace スキャンでトップレベル `hooks` の値の範囲だけを特定し、そこだけテキスト置換するミニスプライサを書いた
- **symlink を保全する**。dotfiles 管理(GNU stow 等)で settings.json がシンボリックリンクになっているケースでは、素朴な「tmp に書いて rename」がリンクを実ファイルに置き換えて破壊する。リンク先を解決してから書き込む
- パースできないファイル(壊れた JSON)は**触らずにエラーにする**。書き込み前に `.bak` バックアップを作る
- hook コマンドには `$HOME` ベースの相対表現を使い、端末固有の絶対パスを設定ファイルに残さない

このあたりは機能というより信頼の問題で、「勝手に settings.json を書き換えるアプリ」が受け入れられるかどうかの分水嶺だと思っています。

## こまごまとした macOS ネイティブ実装の話

本筋ではないので駆け足で。

- フローティングウィンドウは `NSPanel`(`level = .floating`、全 Space 表示)。SwiftUI を `NSHostingView` で載せるハイブリッド構成
- 自動表示/非表示は「waiting/done の**セッションID集合のエッジ**」で判定。件数で判定すると、ユーザーが手動で閉じた後のポーリングで勝手に復活する
- セッションは猫としてピクセルアートの部屋に住んでいて、`session_id` の FNV-1a ハッシュから毛色・歩速・アニメ位相を決定論的に生成。同じセッションはいつ見ても同じ猫になる(乱数を保存せず種から再構築)
- ドット絵は SwiftUI Canvas + Core Graphics で、アンチエイリアスと補間を完全に切って 1 論理 px を矩形塗りでベイク
- 8Hz のシミュレーションだが、スナップショットが前回と等値なら再描画をスキップ。ウィンドウが完全に隠れているときは(オクルージョン通知で)tick 自体を止める

このへんの話はそれぞれ1本記事が書ける分量があるので、反応があれば続編を書きます。

## まとめ

- Claude Code の並列運用は「待たれていることに気づくまでの時間」がボトルネック
- hook → ローカル JSON → FSEvents というファイルベース構成は、単純なわりに壊れにくくてデバッグしやすい
- 「waiting」の判定は Notification の種類分岐+`PreToolUse` の特別扱い+サブエージェント追跡の合わせ技
- ターミナルジャンプに銀の弾丸はない。ホストごとの各個撃破+「誤爆するなら飛ばない」

AgentManager は macOS 13+(Apple Silicon / Intel)対応、7日間無料トライアルがあります(カード登録不要)。並列セッションの「待ちぼうけ」に心当たりがあれば、試してみてください。
